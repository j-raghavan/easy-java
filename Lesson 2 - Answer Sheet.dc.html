Under the Hood: Answer Key

📌 For the student: Attempt all three tasks in the live editor before reading. Pay special attention to integer division and modulo — they appear on almost every AP CS A exam.

1 Memory math

Store 3 int variables (4 bytes each) and print the total bytes used.

int bytesPerInt = 4; int count = 3; int totalBytes = bytesPerInt * count; System.out.println("Total memory: " + totalBytes + " bytes");
Expected outputTotal memory: 12 bytes
Key idea: each variable reserves a fixed-size box in RAM. An int is 4 bytes in Java, so 3 of them use 4 × 3 = 12 bytes. This is literally how the JVM reserves memory when your program starts.

2 Integer division & remainder

30 students, teams of 4. Print full teams (/) and leftover students (%).

int students = 30; int teamSize = 4; int fullTeams = students / teamSize; // integer division int leftOver = students % teamSize; // modulo = remainder System.out.println("Full teams: " + fullTeams); System.out.println("Left over: " + leftOver);
Expected outputFull teams: 7 Left over: 2
Key idea: 30 / 4 is 7, not 7.5 — integer division throws away the decimal. 30 % 4 gives the leftover, 2 (because 7 × 4 = 28, and 30 − 28 = 2). Together, division and modulo split a number into "how many groups" and "what's left."

Common mistake: expecting 30 / 4 to be 7.5. It's only 7.5 if at least one value is a double (e.g. 30.0 / 4).

3 Cause a crash, then fix it

Trigger a runtime error (divide by zero), then fix it. Label the error types.

// BEFORE — this compiles fine, but crashes at runtime: int total = 100; int groups = 0; System.out.println(total / groups); // 💥 ArithmeticException: / by zero
// AFTER — change groups to a non-zero number: int total = 100; int groups = 5; System.out.println(total / groups); // runs fine
Expected output (after fix)20
Labelling the errors: dividing by zero is a runtime error — the code is grammatically correct, so it compiles and starts, then crashes mid-run with an ArithmeticException. By contrast, forgetting a semicolon or misspelling System would be a compile-time error — javac refuses to build it and the program never starts.
